法則 A44

遅延処理 — 物理操作の代わりにフラグ管理

「実際に動かす」のはコストが大きい。代わりに方向や順序だけを記録

「数列を反転する」「2 つの行を入れ替える」のような操作を毎回物理的に行うとO(N) かかります。代わりに フラグや順序配列だけを更新 して、最後に出力時に解釈すると O(1)。

代表的なテクニック:

  • 反転フラグ (A44 本問)
  • インデックス配列で行の順序を持つ (B44)
  • 遅延セグメント木 (上級)

「物理的に動かすのは最後だけ」という発想。

Step 1: 状態を直接持たず、操作の記録だけ持つ

反転操作なら 「反転されているか?」のブール 1 個 を持つ。操作 K 回 → ブール反転 K 回 → 出力時に解釈。

flowchart LR
  A[反転操作 1] --> F[flag = True]
  F --> B[反転操作 2]
  B --> F2[flag = False]

Step 2: アクセス時にフラグを参照

「位置 i にアクセス」のとき、フラグに応じてインデックスを変換 する。real_i = (N - 1 - i) if flag else i。

flowchart LR
  Q["query 位置 i"] --> F{flag?}
  F -- True --> R1["real_i = N-1-i"]
  F -- False --> R2["real_i = i"]

Step 3: 行入れ替えの例

B44 のように行入れ替えは インデックス配列 idx[1..N] を持ち、行 a, b の入れ替えを idx[a], idx[b] = idx[b], idx[a] だけで実現。実際の行データはコピーしない → O(1)。

詰まりやすい点

  • フラグ管理は 読み書きの両方 でフラグ参照を忘れずに。
  • 複数フラグが絡むと管理が破綻しがち。状態の意味 をコメントに明記する。
  • 遅延処理は 最終出力時 にも正しく解釈する必要がある。
